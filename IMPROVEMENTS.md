# Спецификация: underground-psy — Доработки платформы

## Инструкция для Claude Code

Этот документ — спецификация доработок для проекта **underground-psy** (монолит на Laravel). Используй его как источник истины при реализации.

**Если ты обнаружишь расхождение между существующим кодом и требованиями этой спецификации — не исправляй молча. Остановись и задай уточняющий вопрос:** что является правильным — текущий код или спецификация. Возможно, спецификация устарела или код реализует сознательное отклонение.

Этот документ дополняет основную спецификацию троек (spec-triads.md). При конфликтах — этот документ приоритетнее, так как он более свежий.

## 1. Верификация диплома — жёсткая модерация

### Суть изменения

Публикация профиля психолога (`is_published = true`) требует `diploma_verified = true`. Без верифицированного диплома психолог не может пользоваться платформой.

### Поток регистрации психолога

1. Психолог заполняет форму регистрации: имя, email, пароль, роль (психолог), пол
2. На этом же шаге загружает: скан/фото диплома + номер диплома + год выдачи + учебное заведение
3. Подтверждает email (письмо с ссылкой — параллельно с проверкой диплома)
4. Аккаунт создаётся в базе со статусом `pending_verification`
5. Психолог может залогиниться → видит дашборд с сообщением:
   - «Ваш диплом на проверке. Срок проверки — до 2 недель.»
   - Контактная информация на случай, если проверка заняла дольше
6. Доступ ко всей функциональности платформы заблокирован
7. Админ/модератор проверяет диплом → одобряет или отклоняет с комментарием
8. Одобрен → статус аккаунта меняется на `active`, психолог получает уведомление (колокольчик + email), платформа разблокирована
9. Отклонён → психолог видит причину отклонения, загружает новый скан, цикл повторяется. Лимита попыток нет

### Регистрация клиента

Без изменений. Клиент регистрируется свободно: имя, email, пароль, роль (клиент). Никакой верификации диплома. Подтверждение email обязательно.

### Изменения в модели данных

**Таблица: users — изменения:**

| Поле | Тип | Описание |
|------|-----|----------|
| status | enum: active, pending_verification | Статус аккаунта. Клиенты сразу active (после email-верификации). Психологи — pending_verification до одобрения диплома |
| gender | enum: male, female, not_specified | Пол. Заполняется при регистрации |
| email_verified_at | datetime, nullable | Стандартное поле Laravel для MustVerifyEmail |

**Таблица: psychologist_profiles — изменения/уточнения:**

| Поле | Тип | Описание |
|------|-----|----------|
| diploma_scan_url | string | Путь к загруженному файлу (обязательно) |
| diploma_number | string | Номер диплома |
| diploma_year | integer | Год выдачи |
| diploma_institution | string | Учебное заведение |
| diploma_verified | boolean, default false | Флаг одобрения |
| diploma_rejection_comment | text, nullable | Комментарий при отклонении |
| city | string, nullable | Город (для офлайн-фильтра) |

### Бизнес-правила

- `is_published = true` невозможен без `diploma_verified = true`
- Пока `users.status = pending_verification` — доступ ко всей платформе заблокирован (middleware)
- При повторной загрузке диплома (после отклонения): `diploma_verified` сбрасывается в `false`, `diploma_rejection_comment` очищается, `users.status` возвращается в `pending_verification`
- Загруженные файлы: изображения (jpg, png) и PDF
- Проверка диплома — только ручная (админ/модератор)

### State machine — User status (психолог)

```
[pending_verification] → диплом одобрен → [active]
[active] → загрузил новый диплом (если предыдущий отклонён) → [pending_verification]
```

### Уведомления

| Тип | Когда | Кому | Каналы |
|-----|-------|------|--------|
| diploma_approved | Диплом одобрен | Психологу | Колокольчик + email |
| diploma_rejected | Диплом отклонён | Психологу | Колокольчик + email |

## 2. can_consult — допуск к консультациям

### Суть

Психолог должен регулярно посещать интервизии, чтобы быть допущенным к консультациям с клиентами. Если активности недостаточно — психолог скрывается из каталога, новые записи невозможны, но существующие сохраняются.

### Логика расчёта

- **Rolling window:** последние 30 дней
- **Минимум:** 3 интервизии за последние 30 дней
- **Минимальное количество настраивается** в `intervision_settings` (одно число на всю платформу, изменяется только админом)
- **Пересчёт:** в реальном времени — при каждом запросе каталога и при посещении интервизии

### Первый месяц после регистрации

Новый психолог — `can_consult = false`. Сначала набирает минимум интервизий за 30 дней, только потом допускается к консультациям.

### Что блокирует `can_consult = false`

- Психолог **скрыт из каталога** для клиентов
- Психолог **не видит клиентов** (раздел консультаций недоступен для новых записей)
- Существующие записи **сохраняются** и проводятся
- Интервизии и тройки — **доступны** всегда

### Что видит психолог

На дашборде — уведомление с деталями:
- «Вы посетили 1 из 3 обязательных интервизий за последние 30 дней. Для допуска к консультациям посетите ещё 2.»
- Прогресс обновляется в реальном времени

### Метрики в профиле

Счётчики (видны всем):
- Количество троек (по ролям: терапевт / клиент / наблюдатель + общее)
- Количество интервизий
- Количество консультаций (только проведённые через платформу с видеосвязью)
- Стаж на платформе

## 3. OAuth — авторизация через ВК и Яндекс

### Суть

Альтернативный способ входа только для психологов. Клиентам OAuth недоступен — конфиденциальность.

### Реализация

- Laravel Socialite + community-провайдеры для ВК и Яндекс
- Email при OAuth **не обязателен** (ВК может не отдать)
- Если психолог регистрируется через OAuth — всё равно проходит загрузку диплома и верификацию
- Привязка соцсети к существующему аккаунту — **backlog**, не делаем сейчас

### Поток OAuth-регистрации психолога

1. Нажимает «Войти через ВК / Яндекс»
2. Авторизуется у провайдера
3. Возвращается на платформу → попадает на шаг загрузки диплома (имя подтягивается из OAuth)
4. Загружает диплом → аккаунт создаётся со статусом `pending_verification`
5. Далее — стандартный поток верификации

### Изменения в модели данных

Поля `provider` и `provider_id` уже существуют в таблице `users`. Изменений в схеме не требуется.

## 4. Email-верификация

### Суть

Обязательное подтверждение email для обеих ролей (клиент и психолог).

### Реализация

- Включить `MustVerifyEmail` на модели User
- Добавить middleware `verified` на все защищённые маршруты
- Для психологов: email-верификация происходит **параллельно** с проверкой диплома. Психолог может подтвердить email, пока ждёт одобрения диплома
- Для клиентов: после регистрации — письмо с подтверждением. Пока не подтвердил — доступ заблокирован

### Порядок для психолога

1. Регистрация + загрузка диплома
2. Получает письмо для подтверждения email
3. Подтверждает email (в любой момент)
4. Ждёт одобрения диплома
5. Оба условия выполнены (email подтверждён + диплом одобрен) → платформа разблокирована

### Порядок для клиента

1. Регистрация
2. Получает письмо для подтверждения email
3. Подтверждает → платформа доступна

## 5. Транзакционные email-уведомления

### Суть

Только транзакционные письма, привязанные к действиям. Маркетинговых рассылок нет. Отписка не нужна.

### Список транзакционных писем

| Письмо | Когда | Кому |
|--------|-------|------|
| email_verification | При регистрации | Всем |
| diploma_approved | Диплом одобрен | Психологу |
| diploma_rejected | Диплом отклонён с комментарием | Психологу |

| slot_cancelled | Слот тройки отменён | Участникам |

### Реализация

- Laravel Notifications (mail channel)
- Шаблоны в Blade (Markdown mail)
- Очередь (queue) для отправки — не блокировать запрос пользователя

## 6. Роли и права доступа

### Три уровня ролей

| Роль | Описание | Назначение |
|------|----------|-----------|
| ADMIN | Суперадмин (владелец платформы) | Единственный. Вручную в БД или через seeder |
| MODERATOR | Волонтёр-модератор | Назначается ADMIN через админку |
| PSYCHOLOGIST | Психолог | Регистрация + верификация |
| CLIENT | Клиент | Свободная регистрация |

### Права модератора

**Может:**
- Проверка дипломов (одобрить / отклонить с комментарием)
- Модерация комментариев (удалить, бан/разбан на комментирование)
- Модерация заданий для троек (одобрить / отклонить)

**Не может:**
- Назначать роли
- Менять настройки платформы (intervision_settings и т.д.)
- Управлять контентом (если будет CMS)
- Видеть финансовую информацию
- Удалять аккаунты

### Права админа

Всё, что может модератор, плюс:
- Назначение/снятие роли MODERATOR через админку
- Изменение настроек платформы
- Управление справочниками (методы, специализации, языки)
- Полный доступ ко всем разделам

### API

> **PUT /api/admin/users/{id}/make-moderator** — назначить модератором
> Auth: required (role: ADMIN)
> Response 200: { role: "MODERATOR" }

> **PUT /api/admin/users/{id}/remove-moderator** — снять роль модератора
> Auth: required (role: ADMIN)
> Response 200: { role: "PSYCHOLOGIST" }

### Реализация

- Laravel Gates / Policies
- Middleware для проверки ролей на группах маршрутов

## 7. Каталог психологов — фильтры и сортировка

### Фильтры

Все фильтры комбинируются через AND.

| Фильтр | Тип | Источник данных |
|--------|-----|----------------|
| По специализации / проблеме | Множественный выбор | problem_types (M:M) |
| По методу работы | Множественный выбор | methods (M:M) — новый справочник |
| По цене | Диапазон (от / до) | psychologist_profiles.hourly_rate_min / hourly_rate_max |
| По имени | Текстовый поиск | users.name (LIKE) |
| По полу | Одиночный выбор | users.gender |
| По формату работы | Одиночный выбор | psychologist_profiles.work_format (online / offline / both) |
| По городу | Текстовый поиск или выбор | psychologist_profiles.city (для офлайн) |
| По языку | Множественный выбор | languages (мультиселект в профиле) — новое поле |

### Сортировки

| Сортировка | Описание |
|-----------|----------|
| По активности (дефолт) | Взвешенная: активность за последние 90 дней — коэф. 1.0, старше 90 дней — коэф. 0.3. Активность = тройки + интервизии + консультации |
| По количеству консультаций | Только проведённые через платформу |
| По стажу на платформе | Дата регистрации |
| По цене | hourly_rate_min, возрастание |

### Условия показа в каталоге

Психолог виден в каталоге только если ВСЕ условия выполнены:
1. `diploma_verified = true`
2. `is_published = true`
3. `can_consult = true` (минимум 3 интервизии за последние 30 дней)
4. `users.status = active`

### Формула взвешенной активности

```
activity_score =
  (triads_last_90_days + intervisions_last_90_days + consultations_last_90_days) * 1.0
  + (triads_older + intervisions_older + consultations_older) * 0.3
```

Пересчитывается при запросе каталога (или кэшируется с TTL, если будет проблема с производительностью).

### Изменения в модели данных

**Новая таблица: methods** (справочник методов работы)

| Поле | Тип | Описание |
|------|-----|----------|
| id | PK, bigint, auto | |
| name | string | Название метода (ACT, КПТ, гештальт и т.д.) |
| created_at | timestamp | |
| updated_at | timestamp | |

**Новая таблица: psychologist_methods** (M:M)

| Поле | Тип | Описание |
|------|-----|----------|
| id | PK, bigint, auto | |
| psychologist_profile_id | FK → psychologist_profiles.id | |
| method_id | FK → methods.id | |

**Новые поля в psychologist_profiles:**

| Поле | Тип | Описание |
|------|-----|----------|
| work_format | enum: online, offline, both | Формат работы |
| city | string, nullable | Город (для офлайн) |

### Управление справочником методов (админка)

> **GET /api/admin/methods** — список методов
> Auth: required (role: ADMIN)

> **POST /api/admin/methods** — добавить метод
> Auth: required (role: ADMIN)

Видимые метрики:

- Специализации

- Диапазон цены

- Город (если офлайн)
- Языки
- Количество троек (общее)
- Количество интервизий
- Количество консультаций

## 8. Обновлённый поток регистрации

### Психолог

```
1. Форма регистрации:
   - Имя, email, пароль, пол
   - Скан диплома (файл)
   - Номер диплома, год выдачи, учебное заведение
   - [или] OAuth через ВК / Яндекс → имя подтягивается, далее загрузка диплома

2. Аккаунт создаётся: status = pending_verification

3. Параллельно:
   - Письмо для подтверждения email → пользователь подтверждает
   - Админ/модератор проверяет диплом

4. Оба условия выполнены:
   - email_verified_at IS NOT NULL
   - diploma_verified = true
   → status = active, платформа разблокирована

5. Диплом отклонён:
   - Психолог видит причину на дашборде + получает email
   - Загружает заново → цикл повторяется
```

### Клиент

```
1. Форма регистрации: имя, email, пароль
2. Аккаунт создаётся: status = active
3. Письмо для подтверждения email
4. Пока email не подтверждён — доступ заблокирован (middleware `verified`)
5. Email подтверждён → платформа доступна
```

## 9. Границы scope — доработки

### In scope (делаем)

- Жёсткая верификация диплома (загрузка через UI, ручная модерация, блокировка без диплома)
- Статус аккаунта `pending_verification` / `active`
- can_consult: rolling window 30 дней, минимум 3 интервизии, блокировка каталога
- OAuth для психологов (ВК, Яндекс) через Laravel Socialite
- Email-верификация для обеих ролей (MustVerifyEmail)
- Транзакционные email-уведомления
- Уведомления на платформе (колокольчик)
- Роль MODERATOR: назначение через админку, права на одобрения и модерацию
- Справочник методов работы (админка)
- Фильтры каталога: специализация, метод, цена, имя, пол, формат, город, язык
- Сортировки каталога: по активности (взвешенная), консультациям, стажу, цене
- Поля профиля: work_format, city, languages
- Метрики в карточке психолога

### Out of scope

- Привязка соцсети к существующему аккаунту
- Маркетинговые рассылки
- Автоматическая проверка номера диплома
- Отписка от email-уведомлений (транзакционные — не требуется)

## 10. Приоритизация

### До запуска (блокеры)

1. Email-верификация (MustVerifyEmail + middleware `verified`)
2. Загрузка диплома через UI
3. Жёсткая связка: `is_published` требует `diploma_verified`
4. Статус `pending_verification` + блокировка доступа
5. Поток одобрения/отклонения диплома с уведомлениями
6. can_consult — блокировка каталога при недостатке интервизий
7. Роль MODERATOR с правами

### Первое обновление после запуска

8. OAuth (ВК, Яндекс)
9. Фильтры каталога (полный набор)
10. Сортировка по взвешенной активности
11. Справочник методов + M:M
12. Транзакционные email-уведомления (кроме email-верификации — она до запуска)

### Backlog

- Привязка соцсети к существующему аккаунту
- Кэширование activity_score
- Аудит действий админа/модератора
- Artisan-команда для назначения ADMIN

## 11. Технические заметки

- **MustVerifyEmail:** включить на модели User, добавить middleware `verified` + middleware `EnsureAccountActive` (проверяет `status = active`). Два middleware работают в связке: `verified` проверяет email, `EnsureAccountActive` проверяет статус аккаунта.
- **Загрузка диплома:** хранение в `storage/app/private/diplomas/{user_id}/` (не public — только админ видит). Принимать jpg, png, pdf. Максимум 10MB.
- **can_consult:** вычисляется динамически. Query: `SELECT COUNT(*) FROM intervision_attendances WHERE user_id = ? AND attended_at >= NOW() - INTERVAL 30 DAY`. Сравнивается с `intervision_settings.min_sessions`. Можно оформить как scope на модели или как отдельный сервис.
- **Взвешенная активность:** для производительности — если каталог будет тормозить, вынести в отдельную таблицу `psychologist_activity_scores` и пересчитывать по cron (раз в час). На старте — считать на лету.
- **OAuth:** Laravel Socialite. Провайдеры: `socialiteproviders/vkontakte` и `socialiteproviders/yandex`. При OAuth-регистрации психолога — создать User + редирект на шаг загрузки диплома.
- **Транзакционные письма:** Laravel Notifications с mail channel. Класс на каждый тип уведомления. Отправка через очередь (`ShouldQueue`).
