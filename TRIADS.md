# Спецификация: Психотерапевтические тройки

## Инструкция для Claude Code

Этот документ — спецификация фичи «Психотерапевтические тройки» для проекта **underground-psy** (монолит на Laravel). Используй его как источник истины при реализации.

**Если ты обнаружишь расхождение между существующим кодом и требованиями этой спецификации — не исправляй молча. Остановись и задай уточняющий вопрос:** что является правильным — текущий код или спецификация. Возможно, спецификация устарела или код реализует сознательное отклонение.

---

## 1. Краткое описание

Психотерапевтические тройки — формат практики, в котором три психолога разыгрывают терапевтическую сессию в ролях **терапевт**, **клиент** и **наблюдатель**. Каждому участнику выдаётся задание (инструкция) под его роль. Сессия проходит через встроенный видеозвонок. После сессии участники обмениваются устным фидбеком. Участие отображается в профиле как счётчик по ролям.

---

## 2. Проблема и контекст

**Проблема:** у психологов (особенно начинающих) нет доступной безопасной среды, где можно регулярно практиковать терапевтические техники с коллегами и получать обратную связь.

**Текущий workaround:** организация троек через чаты учебных групп, при учебных заведениях — неудобно, нет единой точки входа.

**Целевая аудитория:** практикующие и начинающие психологи/психотерапевты с высшим психологическим образованием или профессиональной переподготовкой.

**Контекст платформы:** underground-psy — площадка для психологов. Помимо троек на платформе будут интервизионные группы и возможность клиентам получать консультации от начинающих психологов. Эти фичи — за скоупом данной спецификации.

## 3. User Stories

### US-1: Создание открытого слота

> Как психолог, я хочу создать открытый слот с заданием, чтобы коллеги могли записаться и мы провели тройку.

**Критерии приёмки:**
- Автор выбирает задание из банка одобренных заданий
- Автор фиксирует свою роль (терапевт / клиент / наблюдатель)
- Автор выбирает дату и время начала
- Автор выбирает режим видимости инструкций: открытый или слепой
- Длительность определяется заданием (не выбирается отдельно)
- Слот появляется в публичной ленте со статусом `open`
- Нельзя создать слот, пересекающийся по времени с другим слотом автора

### US-2: Запись на открытый слот

> Как психолог, я хочу записаться на открытый слот, выбрав свободную роль, чтобы попрактиковаться.

**Критерии приёмки:**
- Пользователь видит ленту слотов с фильтрацией (по дате, свободным ролям, статусу)
- Пользователь выбирает свободную роль и записывается
- Нельзя записаться, если есть пересечение по времени с другим слотом
- Нельзя записаться менее чем за 1 час до начала
- При конкурентной записи на одну роль — один получает место, второй видит ошибку «роль уже занята»
- Пользователь может отписаться от слота — все участники получают уведомление

### US-3: Приглашение конкретных коллег

> Как психолог, я хочу пригласить конкретных коллег на тройку с предложением ролей и времени.

**Критерии приёмки:**
- Автор создаёт приватный слот (visibility: private)
- Автор приглашает конкретных пользователей, предлагая каждому роль
- Приглашённый получает уведомление и может принять или отклонить
- Переговоры по ролям НЕ поддерживаются — только принять / отклонить
- Если все приглашённые отклонили — слот закрывается (cancelled)
- Автор может пригласить другого человека на место отклонившего
- Автор может сделать приватный слот публичным
- Конфликт по времени проверяется на этапе принятия приглашения

### US-4: Проведение сессии

> Как психолог, я хочу в назначенное время подключиться к видеозвонку и провести сессию по заданию.

**Критерии приёмки:**
- Кнопка «Подключиться» доступна за 5 минут до начала
- Подключение первого участника переводит слот в статус `in_progress`
- Каждый участник видит свою инструкцию по роли
- В слепом режиме: терапевт НЕ видит инструкцию клиента; клиент видит свою инструкцию; наблюдатель видит все инструкции
- Если участник не пришёл — оставшиеся могут перераспределить роли через интерфейс
- Ограничение на перераспределение: роли терапевта и клиента обязательны (наблюдатель опционален)
- По истечении длительности — уведомление о скором завершении. Звонок НЕ обрывается автоматически
- После окончания сессии — каждый участник подтверждает, что сессия состоялась
- Все участники должны быть онлайн весь период, иначе не засчитывается
- Когда все активные участники подтвердили — счётчик +1 каждому
- Если хотя бы один не подтвердил — не засчитывается никому

### US-5: Счётчик в профиле

> Как психолог, я хочу видеть в профиле счётчик своих троек по ролям, чтобы отслеживать активность.

**Критерии приёмки:**
- В профиле отображается: количество сессий как терапевт, как клиент, как наблюдатель, общее число
- Счётчик увеличивается только после подтверждения всеми участниками
- Без оценок и рейтингов

### US-6: Предложение задания пользователем

> Как психолог, я хочу предложить своё задание для банка, чтобы расширить набор кейсов.

**Критерии приёмки:**
- Пользователь заполняет форму: название, описание кейса, инструкция для клиента, инструкция для терапевта, инструкция для наблюдателя, длительность
- Задание отправляется на модерацию (статус `pending`)
- Пользователь видит список своих заданий со статусами: pending / approved / rejected
- При отклонении — виден комментарий модератора
- Пользователь может отредактировать отклонённое задание и отправить повторно

### US-7: Модерация заданий (админ)

> Как админ, я хочу модерировать предложенные задания — одобрять, отклонять или редактировать.

**Критерии приёмки:**
- Админ видит очередь заданий со статусом `pending`
- Может одобрить → задание появляется в банке
- Может отклонить с комментарием → автор получает уведомление
- Может отредактировать задание перед одобрением

## 4. Бизнес-правила

### Слоты

- **Минимум для проведения сессии:** терапевт + клиент. Наблюдатель опционален.
- **Дедлайн записи:** за 1 час до начала. После дедлайна запись на слот закрыта.
- **Автоотмена:** если к дедлайну нет минимума (терапевт + клиент) — слот автоматически отменяется.
- **Блокировка пересечений:** пользователь не может быть участником двух слотов, пересекающихся по времени. Проверка при создании слота, записи и принятии приглашения.
- **Одна роль — один человек:** в рамках слота каждая роль может быть занята только одним участником.
- **Один человек — один слот:** пользователь не может записаться на слот дважды.

### Приглашения

- Только принять / отклонить (без переговоров по ролям).
- Если все приглашённые отклонили — слот переходит в `cancelled`.
- Автор может дослать приглашение другому пользователю или сделать слот публичным.

### Видеозвонок

- Подключение доступно за 5 минут до начала.
- Переход в `in_progress` — при подключении первого участника.
- Уведомление о скором завершении по истечении длительности. Автоотключения нет.
- Перераспределение ролей доступно в `in_progress` с ограничением: терапевт + клиент обязательны.

### Подтверждение сессии

- После окончания (по таймеру длительности) каждый участник может подтвердить сессию.
- Все активные участники (status = active) должны подтвердить.
- Если хотя бы один не подтвердил — не засчитывается никому.
- Подтверждение увеличивает счётчик в профиле.

### Задания

- Банк заданий ведёт админ.
- Пользователи могут предлагать задания → модерация.
- Длительность сессии фиксирована в задании.
- Задание содержит отдельные инструкции для каждой роли.

### Режимы видимости инструкций

- **Открытый режим:** все участники видят все инструкции.
- **Слепой режим:** терапевт НЕ видит инструкцию клиента. Клиент видит свою. Наблюдатель видит всё.
- Режим выбирается создателем слота.

## 6. State Machines

### Slot

```
[open] → все роли заняты → [full]
[open] → дедлайн (за 1 час), есть терапевт + клиент → [full]
[open] → дедлайн, нет минимума (терапевт + клиент) → [cancelled]
[open] → автор отменил → [cancelled]
[full] → первый участник подключился → [in_progress]
[in_progress] → все участники подтвердили завершение → [completed]
[in_progress] → (не все подтвердили — слот остаётся in_progress, счётчик не увеличивается)
[cancelled] — конечное состояние
[completed] — конечное состояние
```

### Invitation

```
[pending] → принял → [accepted] → создаётся запись в slot_participants
[pending] → отклонил → [declined]
```

### Task

```
[draft] → отправлено на модерацию → [pending]
[pending] → одобрено → [approved]
[pending] → отклонено → [rejected]
[rejected] → исправлено, отправлено повторно → [pending]
```

### Slot Participant (status)

```
[active] → участник отписался → [cancelled]
```

## 7. API-контракты

### 7.1 Задания

#### GET /api/tasks
Список одобренных заданий (для выбора при создании слота).

- **Auth:** required
- **Query:** `?search=...&page=1`
- **Response 200:**
```json
{
  "data": [
    {
      "id": 1,
      "title": "Разочарование в себе",
      "description": "Клиент переживает острое разочарование...",
      "duration_minutes": 50
    }
  ],
  "meta": { "pagination": {} }
}
```

#### POST /api/tasks
Предложить задание.

- **Auth:** required
- **Body:**
```json
{
  "title": "string",
  "description": "string",
  "instruction_client": "string",
  "instruction_therapist": "string",
  "instruction_observer": "string",
  "duration_minutes": 50
}
```
- **Response 201:** `{ "id": 1, "status": "pending" }`
- **Response 422:** `{ "errors": { "title": "..." } }`

#### GET /api/tasks/my
Мои предложенные задания со статусами.

- **Auth:** required
- **Response 200:**
```json
{
  "data": [
    { "id": 1, "title": "...", "status": "pending", "moderation_comment": null }
  ]
}
```

#### PUT /api/tasks/{id}
Редактировать отклонённое задание и переотправить.

- **Auth:** required (только автор, только status = rejected)
- **Body:** аналогично POST /api/tasks
- **Response 200:** `{ "id": 1, "status": "pending" }`

### 7.2 Админ — модерация заданий

#### GET /api/admin/tasks?status=pending
Очередь на модерацию.

- **Auth:** required (role: admin)

#### PUT /api/admin/tasks/{id}/approve
Одобрить задание (с возможной редактурой).

- **Auth:** required (role: admin)
- **Body:** `{ "title?": "...", "description?": "...", ... }` — опциональные правки
- **Response 200:** `{ "id": 1, "status": "approved" }`

#### PUT /api/admin/tasks/{id}/reject
Отклонить задание.

- **Auth:** required (role: admin)
- **Body:** `{ "moderation_comment": "string" }`
- **Response 200:** `{ "id": 1, "status": "rejected" }`

### 7.3 Слоты

#### GET /api/slots
Лента доступных слотов с фильтрацией.

- **Auth:** required
- **Query:** `?status=open&date_from=...&date_to=...&role=...&page=1`
- **Response 200:**
```json
{
  "data": [
    {
      "id": 1,
      "task": { "id": 1, "title": "...", "duration_minutes": 50 },
      "starts_at": "2026-03-10T15:00:00Z",
      "visibility": "public",
      "blind_mode": false,
      "creator": { "id": 1, "name": "Анна" },
      "participants": [
        { "role": "therapist", "user": { "id": 1, "name": "Анна" } }
      ],
      "available_roles": ["client", "observer"]
    }
  ],
  "meta": { "pagination": {} }
}
```

#### POST /api/slots
Создать слот.

- **Auth:** required
- **Body:**
```json
{
  "task_id": 1,
  "starts_at": "2026-03-10T15:00:00Z",
  "role": "therapist",
  "visibility": "public",
  "blind_mode": false
}
```
- **Response 201:** `{ "id": 1, "status": "open", ... }`
- **Response 422:** `{ "errors": { "starts_at": "Конфликт с другим слотом" } }`

#### DELETE /api/slots/{id}
Отменить слот (только автор).

- **Auth:** required
- **Response 200:** `{ "id": 1, "status": "cancelled" }`
- **Side effect:** уведомление всем записавшимся участникам

#### POST /api/slots/{id}/join
Записаться на слот.

- **Auth:** required
- **Body:** `{ "role": "client" }`
- **Response 201:** `{ "participant_id": 1, "role": "client" }`
- **Response 409:** `{ "error": "Роль уже занята" }`
- **Response 409:** `{ "error": "Конфликт по времени с другим слотом" }`
- **Response 422:** `{ "error": "Запись закрыта (менее 1 часа до начала)" }`

#### DELETE /api/slots/{id}/leave
Отписаться от слота.

- **Auth:** required
- **Response 200:** `{ "status": "cancelled" }`
- **Side effect:** уведомление автору и остальным участникам

#### POST /api/slots/{id}/connect
Подключиться к видеозвонку.

- **Auth:** required (только активный участник слота)
- **Precondition:** не ранее чем за 5 минут до starts_at
- **Response 200:**
```json
{
  "signaling_channel": "slot.{id}.video",
  "ice_servers": [],
  "instructions": {
    "role": "therapist",
    "text": "Ваша инструкция..."
  }
}
```
- **Side effect:** если слот в статусе `full` — переводит в `in_progress`

#### POST /api/slots/{id}/reassign-role
Перераспределить роль (в in_progress, при неявке участника).

- **Auth:** required (только активный участник)
- **Body:** `{ "new_role": "client" }`
- **Response 200:** `{ "role": "client" }`
- **Response 422:** `{ "error": "Роли терапевта и клиента должны быть заняты" }`

#### POST /api/slots/{id}/confirm
Подтвердить завершение сессии.

- **Auth:** required (только активный участник)
- **Response 200:** `{ "confirmed": true }`
- **Side effect:** если все активные участники подтвердили → слот → `completed`, счётчики +1

### 7.4 Приглашения

#### POST /api/slots/{id}/invite
Пригласить в слот.

- **Auth:** required (только автор слота)
- **Body:** `{ "user_id": 2, "proposed_role": "client" }`
- **Response 201:** `{ "invitation_id": 1, "status": "pending" }`
- **Side effect:** уведомление приглашённому

#### PUT /api/invitations/{id}/accept
Принять приглашение.

- **Auth:** required (только invitee)
- **Response 200:** `{ "participant_id": 1, "role": "client" }`
- **Response 409:** `{ "error": "Конфликт по времени с другим слотом" }`

#### PUT /api/invitations/{id}/decline
Отклонить приглашение.

- **Auth:** required (только invitee)
- **Response 200:** `{ "status": "declined" }`
- **Side effect:** уведомление автору слота

#### PUT /api/slots/{id}/make-public
Сделать приватный слот публичным.

- **Auth:** required (только автор)
- **Response 200:** `{ "visibility": "public" }`

### 7.5 Чат слота

#### GET /api/slots/{id}/messages
Сообщения чата.

- **Auth:** required (только участник слота)
- **Response 200:**
```json
{
  "data": [
    { "id": 1, "user": { "id": 1, "name": "Анна" }, "body": "...", "created_at": "..." }
  ]
}
```

#### POST /api/slots/{id}/messages
Отправить сообщение.

- **Auth:** required (только участник слота)
- **Body:** `{ "body": "string" }`
- **Response 201:** `{ "id": 1, "body": "...", "created_at": "..." }`
- **WebSocket event:** `SlotMessageSent` → канал `slot.{id}.chat`

## 10. Границы scope

### In scope (делаем)
- Банк заданий (CRUD для админа, предложение от пользователей, модерация)
- Создание слота (открытый + приглашение)
- Запись на слот, отмена записи, отмена слота
- Лента слотов с фильтрацией
- Страница «мои слоты» со статусами
- Встроенный видеозвонок (WebRTC mesh, без записи)
- Отображение инструкций по ролям (открытый / слепой режим)
- Перераспределение ролей при неявке
- Подтверждение сессии всеми участниками
- Счётчик в профиле по ролям
- Текстовый чат внутри слота
- Уведомления (колокольчик на платформе, real-time через WebSocket)

### Out of scope (не делаем)
- Переговоры по ролям в приглашениях
- Запись видеозвонков
- Шаринг экрана
- Рейтинги и оценки
- Интервизионные группы
- Регистрация и верификация психологов (отдельная фича)
- Консультации с клиентами
- Push-уведомления (только колокольчик)
- TURN-сервер

## 12. Технические заметки для разработки

- **Миграции:** создать таблицы в порядке зависимостей: tasks → slots → slot_participants → slot_invitations → slot_messages → notifications.
- **Partial unique index:** `slot_participants` требует unique constraint с условием `WHERE status = 'active'`. В PostgreSQL это нативно, в MySQL можно через generated column + unique index.
- **Enum-поля:** в Laravel удобнее через PHP enum (8.1+) + cast. Для миграций — string с валидацией, не нативный MySQL enum (проще менять).
- **Конкурентный доступ:** при записи на роль — использовать pessimistic lock или atomic update с проверкой `WHERE role = ? AND status = 'active'` для предотвращения race condition.
- **Scheduler:** задача на проверку дедлайнов слотов (за 1 час до начала) — каждую минуту или через delayed job.
- **Broadcasting:** настроить каналы как private (авторизация через `Broadcast::channel`), чтобы только участники слота имели доступ к чату и сигналингу.
- **WebRTC:** фронтенд-логика — обмен SDP/ICE через WebSocket-канал `slot.{id}.video`. Бэкенд только пробрасывает сообщения между участниками.
